## 二、底层硬件底座（架构设计第一性原理）

### 2.1 GPU体系深度

#### 2.1.1 NVIDIA架构世代
- **Ampere/Ada/Hopper/Blackwell**：微架构差异、SM单元、Tensor Core、RT Core硬件限制
- **指令集**：FP8/TF32硬件指令集支持
- **显存层级**：HBM2e/HBM3、L2 Cache、片上Shared Memory、显存带宽/位宽/吞吐计算
- **显存墙**：带宽瓶颈识别与优化策略

#### 2.1.2 总线与互联
- **PCIe体系**：PCIe 4.0/5.0通道数、上下行带宽、总线竞争、PCIe隔离、NUMA节点绑定
- **NVLink拓扑**：全连接/环形/树状拓扑、NVLink交换、多GPU通信物理约束
- **GPUDirect**：GPU直接通信、RDMA绕过CPU

#### 2.1.3 GPU虚拟化
- **MIG硬隔离**：GPU切分、算力/显存硬隔离、隔离级故障域
- **MPS软共享**：多进程GPU复用、显存抢占、QoS限制、调度冲突
- **vGPU**：时间片切分、显存QoS限制

#### 2.1.4 异构算力
- **AMD GPU/昇腾NPU/昆仑芯/摩尔线程**：硬件差异与统一纳管难点
- **驱动生态**：设备插件开发、通信库适配

### 2.2 CPU & 内存体系

#### 2.2.1 CPU架构
- **x86/ARM差异**：大核/小核调度、CPU睿频/功耗限制
- **NUMA架构**：多NUMA节点、跨NUMA访问延迟、本地内存/远端内存性能差
- **CPU-GPU亲和拓扑**：CPU直连GPU vs 南桥转接GPU性能差异

#### 2.2.2 内存技术
- **DDR4/DDR5**：内存带宽与延迟
- **大页内存（HugePage）**：预留配置、性能优化
- **透明大页THP**：自动大页管理、性能权衡
- **锁内存mlock**：防止内存交换、保障AI大内存进程

#### 2.2.3 总线拓扑
- 多卡拓扑冲突排查
- 硬件BMC管理、风扇/功耗/温度风控

### 2.3 服务器硬件与运维

#### 2.3.1 AI服务器整机架构
- **训练机型**：高GPU配比
- **推理机型**：高密度部署
- **预处理机型**：大核高IO

#### 2.3.2 硬件故障域设计
- **分层容灾**：单机故障 → 机架故障 → 机柜故障 → 机房故障
- **冗余设计**：电源冗余、风扇散热、硬件BMC管理
- **硬件生命周期**：IPMI、硬件健康巡检、硬盘/GPU坏件预测

## 三、Linux高性能内核（架构师底层底牌）

#### 3.1.1 调度器
- **CFS调度器**：完全公平调度、虚拟运行时间

#### 3.1.3 Cgroup v1/v2
- **全量子系统**：cpu、cpuset、memory、blkio、devices、hugetlb
- **资源限制**：软限制/硬限制、资源抢占、OOM控制

## 四、分布式系统理论（架构设计核心约束）

### 4.1 基础理论
- **CAP理论**：一致性、可用性、分区容错性权衡
- **BASE理论**：基本可用、软状态、最终一致性
- **最终一致性 vs 强一致性**：适用场景

### 4.2 分布式核心问题
- **网络分区**：分区容错设计
- **时钟漂移**：分布式时间同步
- **分布式超时**：超时设置策略
- **脑裂问题**：Split-Brain防护

### 4.3 分布式共识
- **Raft算法**：选主、日志复制、快照、脑裂防护
- **Paxos算法**：原理与Raft对比
- **etcd实现**：集群选主、日志复制机制

### 4.4 分布式锁
- **etcd强一致锁**：基于Raft的分布式锁
- **Redis锁**：优缺点、Redlock算法
- **AI集群任务竞争控制**

### 4.5 流量与容错
- **限流**：令牌桶/漏桶算法
- **熔断**：故障快速失败
- **降级**：服务降级策略
- **重试**：指数退避重试
- **超时控制**：层级超时设计
- **幂等设计**：防重复处理

### 4.6 负载均衡
- **四层/七层负载均衡**：L4/L7差异
- **一致性哈希**：节点增减影响最小化
- **本地优先调度**：减少跨节点通信
- **亲和调度**：任务与数据亲和

### 4.7 AI场景专属分布式约束
- **分布式训练梯度同步一致性**：同步/异步训练
- **Checkpoint分布式读写一致性**：多节点并发读写锁
- **大规模集群任务分片**：任务分片容错
- **局部故障无感**：故障自愈设计

```bash
# 关闭Swap
swapoff -a
sed -i 's/^UUID.*swap/#&/' /etc/fstab

# 内存调优
echo 0 > /proc/sys/vm/swappiness
echo 1 > /proc/sys/vm/zone_reclaim_mode
echo 262144 > /proc/sys/vm/min_free_kbytes

# 大页内存配置
echo 524288 > /proc/sys/vm/nr_hugepages

# 网络全局调优
cat >> /etc/sysctl.conf << EOF
net.core.somaxconn = 65535
net.ipv4.tcp_tw_reuse = 1
net.core.rmem_max = 16777216
net.core.wmem_max = 16777216
net.ipv4.tcp_rmem = 4096 87380 16777216
net.ipv4.tcp_wmem = 4096 65536 16777216
EOF
sysctl -p
```

## 二十四、Checkpoint机制深度解析

### 24.1 核心定义

> 用**空间换时间**，在计算链路的中间保存部分中间结果（检查点），以减少需要长期保存的激活内存，同时在反向或后续计算时按需重新计算。

```
输入 x → f₁ → f₂ → ... → fₙ → 输出 y
          ↑保存   ↑保存        ↑保存
        检查点(Checkpoint)
```

| | 无检查点 | 有检查点 |
|--|---------|---------|
| | 保存所有中间激活 | 只保存少量检查点 |
| 内存 | ⚠ 内存占用大 O(N) | ✅ 内存占用小 O(√N)~O(log N) |
| 计算 | 低 O(N) | 高 O(N·α)，α>1 |
| 复杂度 | 低 | 中等 |

### 24.2 设计动机

长链路的反向传播需要保存大量中间激活，成为内存与效率的瓶颈。

**核心权衡**：减少内存占用 ⟷ 增加计算开销

### 24.3 机制分析

以分段检查点为例（每k层保存一个检查点）：

- **前向（保存检查点）**：x → ① → ② → ... → Ⓚ → ... → Ⓝ，仅保存C，丢弃中间激活
- **反向（按需重算）**：从Cₘ₋₁重算到Cₘ，利用结果反向传播

**微观视角**：
- Linear → GELU → Dropout → ...
- 每个算子是确定性的，输入相同则输出相同
- 无需保存中间激活，节省内存

### 24.4 场景行为

| 训练 (Training) | 推理 (Inference) |
|-----------------|------------------|
| **目标**：反向传播更新参数 | **目标**：仅前向计算得到输出 |
| 使用检查点以降低显存占用 | 通常不需要检查点 |
| **流程**：输入 → 前向计算(保存检查点) → 损失计算 → 反向传播(重算+求梯度) → 参数更新 | **流程**：输入 → 前向计算(无需保存) → 输出 |
| **效果**：显著降低显存占用，支持更长序列、更大模型或更大batch size | **效果**：直接运行前向，追求最低延迟与最高吞吐 |

### 24.5 常见策略

- **固定间隔分段**：均匀分段保存检查点
- **自适应分段**：根据内存压力动态调整
- **拓扑感知分段**：按耗时/内存拓扑感知保存
- **与混合精度、显存优化器配合使用**

| 步骤 | 详细介绍 |
|------|---------|
| **选择硬件 (GPU/TPU)** | 根据模型规模和预算选择硬件。小模型用单卡/多卡GPU，大模型用GPU集群或TPU Pod。 |
| **配置分布式框架** | 选择并行策略：DeepSpeed ZeRO（显存优化）、Megatron-LM并行（TP/PP/SP）。配置NCCL通信、节点拓扑。 |
| **启动训练** | 加载数据、初始化模型、设置优化器（AdamW）、学习率调度（Cosine/Warmup）、混合精度（FP16/BF16/FP8）。 |
| **监控训练指标** | 实时跟踪Loss、Learning Rate、Throughput、GPU利用率、显存占用。使用WandB/TensorBoard可视化。 |
| **损失函数收敛？** | 判断Loss是否稳定下降并趋于平稳。**否** → 调整超参数（学习率、Batch Size、权重衰减）；**是** → 保存检查点。 |
| **保存检查点** | 定期保存模型权重、优化器状态、训练配置。支持断点续训，便于后续微调或部署。 |

### 26.3 阶段三：模型优化（可选循环）

| 步骤 | 详细介绍 |
|------|---------|
| **需要优化？** | 评估模型是否满足部署要求（延迟、显存、精度）。**是** → 选择优化策略。 |
| **量化 (INT8/FP16)** | 降低权重/激活精度，减少显存和计算量。PTQ简单快速，QAT精度更高。 |
| **剪枝 (减少参数)** | 移除不重要权重，稀疏化模型。结构化剪枝兼容性好，非结构化剪枝压缩率高但需专用硬件支持。 |
| **蒸馏 (压缩模型)** | 用小模型学习大模型行为，实现知识迁移。可结合量化/剪枝进一步压缩。 |
| **评估模型性能** | 在验证集上测试优化后模型的精度、延迟、吞吐量。确保优化不显著降低效果。 |

### 26.4 阶段四：部署阶段

| 步骤 | 详细介绍 |
|------|---------|
| **准备部署** | 导出模型为部署格式（ONNX/TensorRT Engine/Safetensors），准备推理代码和配置文件。 |
| **选择推理框架** | **vLLM**：内存优化，高吞吐；**TensorRT-LLM**：NVIDIA GPU极致性能；**Triton**：多模型统一管理。 |
| **容器化** | 将模型和推理服务打包为Docker镜像。包含依赖库、运行时、模型权重，确保环境一致性。 |
| **部署到Kubernetes** | 使用K8s管理推理服务Pod。配置GPU资源限制、节点亲和性、健康检查。 |
| **配置KServe** | 使用KServe提供标准化推理接口。支持自动扩缩容（HPA/KPA）、A/B测试、金丝雀发布。 |
| **设置自动缩放** | 根据QPS/延迟/GPU利用率自动调整Pod数量。设置最小/最大副本数，平衡成本与性能。 |

### 26.5 阶段五：运维监控

| 步骤 | 详细介绍 |
|------|---------|
| **配置负载均衡** | 使用Ingress/Nginx/Envoy分发请求。支持加权轮询、最少连接、一致性哈希等策略。 |
| **集成API网关** | 提供统一入口，处理认证鉴权、限流熔断、请求路由、日志记录。代表：Kong、APISIX、Envoy。 |

## 二十七、分布式训练算法路径（五大并行维度）

### 27.1 并行策略总览

| 维度 | 全称 | 关键算法/技术点 |
|------|------|----------------|
| **DP** | Data Parallelism（数据并行） | 分布式数据、DP梯度、分布式Adam、ZeRO-1/2/3、混合精度Adam、IO-load-save |
| **TP** | Tensor Parallelism（张量并行） | col-parallel、row-parallel、SwiGLU、GQA、LMhead、Embedding、RMS-Norm、CrossEntropy |
| **PP** | Pipeline Parallelism（流水线并行） | 梯度检查点、PP-basic、Gpipe、PipeDream、Zero-Bubble、Easy-Dualpipe、DualPipe |
| **CP** | Context Parallelism（上下文并行） | Ring-AllReduce、Softmax、Online-Softmax、Ring-Softmax、FlashAttention-2、Ring-Attention |
| **EP** | Expert Parallelism（专家并行） | All2All、异步All2All、TopK梯度、SMoE、GShard、1F1B Basic、1F1B重叠、V3-MoE结构 |

### 27.2 项目特点

- **纯PyTorch从零实现**：5大类并行算法（DP、TP、PP、CP、EP），不依赖DeepSpeed和Megatron框架
- **硬核手撕关键算法**：
  - Backward手写实现
  - 分布式gradient和ZeRO-ADAM
  - MoE EP 1F1B下的通信-计算重叠
- **Step-by-step手撕重点**：
  - DP: ZeRO-3
  - TP: Llama
  - CP: RingAttention
  - PP: DualPipe
  - EP: Gshard
- **低门槛运行**：不需要多卡环境，纯CPU GLOO backend可运行所有实例，无须Triton和CUDA等基础

## 二、各阶段详细学习路径与推荐仓库

### Phase 1：系统基础与 GPU 体系结构

**核心知识：**
- CUDA 编程模型与内存层次（Global/Shared/Local/Register）
- GPU 架构演进（Volta → Ampere → Hopper → Blackwell）
- NCCL 集合通信原理与拓扑感知
- Linux 内核调度、内存管理、RDMA 网络

**推荐仓库：**

| 仓库 | 说明 |
|:---|:---|
| **[NVIDIA/cuda-samples](https://github.com/NVIDIA/cuda-samples)** | 官方 CUDA 示例，从基础到高级特性全覆盖 |
| **[NVIDIA/nccl](https://github.com/NVIDIA/nccl)** | NVIDIA 集合通信库源码，理解 all-reduce/all-gather 实现 |
| **[NVIDIA/cutlass](https://github.com/NVIDIA/cutlass)** | CUDA C++ 模板库，学习高性能 GEMM/Convolution 模板编程 |
| **[facebookresearch/fairscale](https://github.com/facebookresearch/fairscale)** | Meta 的 PyTorch 扩展，含 FSDP、ZeRO、流水线并行实现 |

**关键实践：** 用 CUTLASS 手写一个优化过的矩阵乘法，性能达到 cuBLAS 的 80% 以上。

## 五、给你的行动建议

1. **从 Triton 入手**：Triton 是连接 Python 与 GPU 硬件的桥梁，掌握了它，你就同时理解了编译器和性能优化 
2. **精读 vLLM 源码**：它是当前 LLM 推理系统的集大成者，理解了 PagedAttention 和 Scheduler，就掌握了现代推理系统的核心设计模式
3. **参与开源贡献**：给 PyTorch、vLLM、DeepSpeed 提 PR，这是最快的成长方式
4. **构建端到端项目**：不要只停留在跑 demo，要完整经历从需求分析→架构设计→性能调优→生产部署的全周期

这条路线通常需要 **3-5 年** 的深耕，但每一步都有明确的能力验证标准。建议你从 Phase 1 和 Phase 2 并行开始，因为编译器思维和系统思维是 AI Infra 架构师的底层操作系统。
