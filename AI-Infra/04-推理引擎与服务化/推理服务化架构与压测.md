## 1. 推理服务化分层架构

```
网关层 → 调度层 → 模型后端
```

### 1.1 网关层

- **统一入口**：认证鉴权、API Key 管理
- **流量控制**：Token 级限流、按用户/模型维度限流
- **请求路由**：根据模型名称/版本路由到对应后端
- **协议转换**：统一 OpenAI-compatible API，多模型格式适配
- **代表组件**：Kong、APISIX、Envoy

### 1.2 调度层

- **负载均衡**：加权轮询、最少连接、一致性哈希（KV Cache 亲和路由）
- **模型调度**：根据 GPU 资源、模型热度、延迟要求分配后端
- **请求排队**：无可用实例时排队等待，超时拒绝
- **灰度发布**：按比例路由到新版本模型，A/B 测试

### 1.3 模型后端

- **推理引擎**：vLLM、SGLang、TensorRT-LLM、llama.cpp
- **模型加载**：启动时加载权重，显存驻留
- **推理执行**：Prefill + Decode、Continuous Batching
- **健康检查**：定期探活，异常自动摘除

## 2. 服务化核心能力

### 2.1 多模型混部

- **资源共享**：一张 GPU 可同时运行多个小模型
- **显存划分**：MIG 硬隔离 或 MPS/显存限制软隔离
- **调度策略**：按模型优先级分配 GPU 资源
- **优势**：GPU 利用率提升，降低推理成本

### 2.2 动态加载/卸载

- **按需加载**：首次请求时加载模型到 GPU 显存
- **LRU 卸载**：长时间无请求的模型自动卸载，释放显存
- **预热机制**：提前加载热点模型，消除冷启动延迟
- **权衡**：加载延迟 vs 显存占用

### 2.3 显存池化管理

- **统一显存池**：KV Cache + 模型权重共享显存空间
- **动态分配**：PagedAttention 物理块按需分配/回收
- **显存复用**：请求结束后 KV Cache 块回收复用
- **碎片整理**：定期压缩 Block Table，减少碎片

### 2.4 限流/熔断/灰度

| 机制 | 说明 | 触发条件 |
|------|------|---------|
| **限流** | 令牌桶/滑动窗口限制请求速率 | QPS 超阈值 |
| **熔断** | 连续失败后快速拒绝，防止级联故障 | 错误率 > 50% |
| **降级** | 非核心功能关闭（如关掉 Prefix Cache） | 显存/算力紧张 |
| **灰度** | 新模型逐步放量（10% → 50% → 100%） | 新版本上线 |
| **A/B 测试** | 同流量对比两版本模型效果 | 模型迭代评估 |

### 2.5 A/B 测试与模型对比

- **流量分割**：网关层按比例分发请求到 A/B 模型
- **指标对比**：延迟分布、Token 生成质量、用户满意度
- **回滚机制**：B 版本效果差时自动切回 A 版本

## 3. 弹性架构设计

### 3.1 自动扩缩容

| 策略 | 原理 | 适用 |
|------|------|------|
| **HPA（水平扩缩）** | 根据 GPU 利用率/QPS/队列长度自动增减 Pod 数 | 在线推理 |
| **KPA（Knative）** | 请求驱动，0→N 弹性伸缩（含缩容到 0） | 低频模型 |
| **VPA（垂直扩缩）** | 不适用 GPU（显存固定） | ❌ |

**扩缩容参数**：
- 最小副本数：保障基础可用性
- 最大副本数：控制成本上限
- 扩容阈值：GPU 利用率 > 80% 或 请求队列 > 10
- 缩容阈值：GPU 利用率 < 30% 持续 5 分钟

### 3.2 高可用容灾

- **多副本部署**：至少 2 副本（跨节点/跨机架）
- **健康检查**：Liveness Probe（Kernel 心跳）、Readiness Probe（模型加载完成）
- **故障转移**：Pod 异常时自动摘除，请求路由到健康副本
- **滚动更新**：新版本逐步替换旧版本，避免服务中断

## 4. 推理性能压测与基准

### 4.1 压测模型

标准化测试方案：

| 测试场景 | 输入长度 | 输出长度 | 并发数 |
|----------|---------|---------|--------|
| **短文本** | 128 tokens | 128 tokens | 1~128 |
| **中等文本** | 1024 tokens | 256 tokens | 1~64 |
| **长文本** | 4096 tokens | 512 tokens | 1~32 |
| **极限长文** | 32K tokens | 1024 tokens | 1~8 |

### 4.2 核心指标体系

| 指标 | 定义 | 目标 |
|------|------|------|
| **QPS** | 每秒完成请求数 | 越高越好 |
| **TTFT（首Token延迟）** | 首个 token 生成时间 | < 200ms（实时场景） |
| **TPOT（每Token延迟）** | 每个 token 平均生成时间 | < 30ms |
| **P50/P95/P99 延迟** | 分位数延迟 | P99 < 2×P50 |
| **吞吐量（tokens/s）** | 每秒生成 token 总数 | 最大化 |
| **显存利用率** | 已分配 / 总显存 | 80-90%（留 10% buffer） |
| **GPU 利用率** | SM 活跃时间占比 | > 80% |
| **请求排队时间** | 请求等待调度的时间 | < 100ms |

### 4.3 瓶颈定位方法论

```
逐层定位：

1. GPU 利用率低（< 60%）
   ↓
2. 检查显存使用 → 碎片多？Batch Size 太小？
   ↓
3. 检查 KV Cache → 命中率低？Block 分配慢？
   ↓
4. 检查网络 → 请求排队？网关延迟？
   ↓
5. 检查调度 → Continuous Batching 暂停频繁？
```

**排查工具**：
- Nsight Systems：GPU 时间线分析
- PyTorch Profiler：算子级耗时
- vLLM metrics endpoint：调度器/显存内部指标
- Prometheus + Grafana：QPS/P99/GPU 利用率趋势

### 4.4 常见优化手段

| 瓶颈 | 优化 |
|------|------|
| GPU 利用率低 | 增大 Batch Size、启用 Continuous Batching |
| KV Cache 碎片多 | 调整 Block Size、启用 Prefix Caching |
| TTFT 过高 | Chunked Prefill、PD 分离 |
| TPOT 过高 | 量化（FP8/INT4）、启用 CUDA Graph |
| 显存不足 | 减少 max_num_seqs、启用量化、降低 Block Size |

## 5. 推理优化进阶

### 5.1 稀疏推理

- **结构化稀疏**：按固定模式裁剪权重（2:4 稀疏），硬件加速
- **非结构化稀疏**：任意位置稀疏，压缩率高但需专用硬件
- **动态稀疏**：MoE 模型中只激活部分专家，天然稀疏
- **效果**：计算量降低 30-50%，延迟相应降低

### 5.2 模型蒸馏

- **原理**：用小模型（Student）学习大模型（Teacher）的输出分布
- **黑盒蒸馏**：只使用 Teacher 的输出 logits
- **白盒蒸馏**：使用 Teacher 的中间层特征
- **适用场景**：
  - 边缘部署：大模型蒸馏到小模型
  - 专项任务：通用大模型蒸馏到领域模型
- **效果**：模型体积缩小 5-10 倍，性能保留 90%+

### 5.3 动态负载限流

- **自适应限流**：根据 GPU 负载动态调整限流阈值
- **优先级排队**：
  - 高优（VIP 用户）：低延迟保障，不排队
  - 中优（普通用户）：正常排队
  - 低优（批量任务）：可能长时间等待
- **过载保护**：
  - 请求队列满 → 拒绝新请求（HTTP 429）
  - 显存不足 → 暂停接收新请求，等待已有请求完成
  - GPU 温度过高 → 降频或暂停部分推理

---

> 🔑 **核心要点**：推理服务化不只是"把模型跑起来"，而是需要网关→调度→后端的完整分层架构。性能压测需要多场景（短/中/长/极限）覆盖，瓶颈定位遵循"GPU→显存→网络→调度"逐层排查方法论。
