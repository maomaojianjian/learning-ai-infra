# 大模型训练到部署完整流程

> 从数据准备到运维监控的五阶段完整流水线，覆盖大模型生命周期全流程。

---

## 阶段一：数据准备

| 步骤 | 说明 |
|------|------|
| **数据收集** | 从互联网、数据库、API 获取原始数据（文本/图像/代码），需考虑版权和合规性 |
| **数据清洗** | 去重（MinHash/SimHash）、过滤（语言检测/质量评分）、敏感信息脱敏 |
| **数据标注** | 分类标签、问答对、指令遵循格式（Instruction Tuning）、偏好数据（RLHF） |
| **数据集划分** | 训练集/验证集/测试集（通常 8:1:1），确保分布一致 |

---

## 阶段二：训练阶段

| 步骤 | 说明 |
|------|------|
| **选择硬件** | 根据模型规模和预算选择 GPU/TPU。小模型用单卡/多卡 GPU，大模型用 GPU 集群或 TPU Pod |
| **配置分布式框架** | DeepSpeed ZeRO（显存优化）、Megatron-LM（TP/PP/SP）。配置 NCCL 通信、节点拓扑 |
| **启动训练** | 加载数据、初始化模型、优化器(AdamW)、学习率调度(Cosine/Warmup)、混合精度(FP16/BF16/FP8) |
| **监控训练指标** | Loss、LR、Throughput、GPU 利用率、显存占用。WandB/TensorBoard 可视化 |
| **判断收敛** | Loss 稳定下降并趋于平稳 → 保存检查点；否则调整超参数 |
| **保存检查点** | 定期保存模型权重、优化器状态、训练配置，支持断点续训 |

---

## 阶段三：模型优化（可选循环）

| 步骤 | 说明 |
|------|------|
| **量化（INT8/FP8）** | 降低权重/激活精度，减少显存和计算量。PTQ 简单快速，QAT 精度更高 |
| **剪枝（减少参数）** | 移除不重要权重。结构化剪枝兼容性好，非结构化剪枝压缩率高 |
| **蒸馏（压缩模型）** | 小模型学习大模型行为，实现知识迁移 |
| **评估性能** | 验证集上测试精度/延迟/吞吐量，确保优化不显著降低效果 |

---

## 阶段四：部署阶段

| 步骤 | 说明 |
|------|------|
| **准备部署** | 导出模型格式（ONNX/TensorRT Engine/Safetensors），准备推理代码和配置 |
| **选择推理框架** | vLLM（高吞吐）、TensorRT-LLM（NVIDIA 极致性能）、Triton（多模型管理） |
| **容器化** | Docker 镜像包含依赖库、运行时、模型权重，确保环境一致性 |
| **部署到 K8s** | 配置 GPU 资源限制、节点亲和性、健康检查 |
| **配置 KServe** | 标准化推理接口，自动扩缩容、A/B 测试、金丝雀发布 |
| **自动缩放** | 根据 QPS/延迟/GPU 利用率自动调整 Pod 数量 |

---

## 阶段五：运维监控

| 步骤 | 说明 |
|------|------|
| **负载均衡** | Ingress/Nginx/Envoy，加权轮询、最少连接、一致性哈希 |
| **API 网关** | 认证鉴权、限流熔断、请求路由、日志记录（Kong/APISIX） |
| **监控告警** | Prometheus 采集 GPU 利用率/显存/QPS、Grafana Dashboard、OpenTelemetry 全链路追踪 |
| **性能分析** | 分析推理瓶颈：算子耗时、内存带宽、通信延迟（Nsight/PyTorch Profiler） |
| **发现瓶颈** | 调整 Batch Size、启用 CUDA Graph、优化 KV Cache 管理 |
| **持续监控** | 7×24 监控，设置告警阈值（P99 延迟/错误率/GPU 温度） |
| **服务稳定** | 持续收集反馈数据，为下一轮 SFT/RLHF 迭代准备 |

---

> 🔑 **核心要点**：大模型的完整生命周期不是线性的，而是循环迭代的。训练 → 优化 → 部署 → 监控 → 反馈数据 → 再训练，形成持续改进的飞轮。AI Infra 架构师需要保障这个飞轮的每一环都高效运转。
